# authlyn-interactive — Code Review (2026-05-29)

## Executive summary

**Verdict: PASS with hardening backlog.** The codebase is mature and the load-bearing security invariants hold: server-side identity from the session cookie, authorization re-derived on every mutate (never from the request body), privacy-404 collapse, parameterized SQL throughout, the path-traversal escape guard on the media store, and a panic-free markup parser. No critical defect was found. The review surfaced **19 report-worthy findings** and rejected **28 candidates** after adversarial verification (the 20 most substantive rejects are written up in Appendix A).

**Counts by severity:** 0 critical · 2 high · 4 medium · 13 low (incl. 6 docs-mismatch). **Rejected (false positives / non-defects): 28.**

**Posture (one honest paragraph).** What is solid: inv1/inv2/inv4 authorization is fundamentally sound — every mutating handler re-derives role from `guild_member` + channel membership + persona ownership; inv8 (no SQL injection) is upheld at every dynamic-query site (only compile-time fragments and loop indices are spliced, all values bound); inv10's core escape guard (canonicalize + `starts_with(media_dir)`) is implemented and anchor-tested; inv12 (parser panic-freedom) was empirically validated against ~3.1M fuzz inputs with zero panics; the disjointness invariant holds in its load-bearing direction (the wasm bundle is free of ssr crates). What concerns: the two high findings are both real exploitable gaps — a **revoked editor can keep authoring under a persona** (the per-channel wear table is never cleared on revoke/leave/delete and the bare-message send path does not re-gate it), and the **media upload accepts arbitrary MIME and reflects it with no `nosniff`** (stored-XSS / content-sniffing on the app's own origin). The medium tier is dominated by **inv13 write-conflict races** on three unwrapped CREATE-against-UNIQUE paths (register, channel-wear, push-subscribe) that intermittently 500 instead of resolving to an idempotent 409 — these reflect a real un-executed-MVCC-reasoning gap (no live SurrealDB was available to confirm error-text routing). The low tier is mostly **untested-but-correct invariants** and orphan-row cascade gaps in the 30d/1d purge. Recurring theme across dimensions: the *code* is generally correct, but **the anchor-test net has holes precisely at the edges** (same-`sent_at` cursor tie-break, concurrent races, reset-flow indistinguishability, body-char-cap, persona-wear revocation), so several invariants are upheld today by inspection but unguarded against regression.

**Scope.** Whole codebase (80 files), 13 review dimensions (D1a–D13), adversarial verification of every candidate. Every surviving finding carries an exact `file:line`, a concrete reachable trigger, the invariant it breaks, and the anchor-test gap that lets it ship.

## Findings by severity

### HIGH

#### F-D1b-1 — `src/server/personas/editors.rs:71-95` — inv3 (persona wear gated by can_edit_persona; authz re-derived on every mutate) — correctness
**Claim.** `remove_editor` revokes an editor's persona access but only clears the deprecated per-guild `guild_member.active_persona`; it never deletes the editor's `channel_active_persona` rows. Because message-send re-checks `can_edit_persona` ONLY for a client-supplied `persona_id` and trusts the stored `channel_active_persona` for the bare-message fallback (resolved unconditionally, never re-gated), a revoked editor keeps wearing and stamping the persona on bare messages in every channel where they had worn it.
**Evidence.** `editors.rs:71-79` does `DELETE FROM persona_editor …` + `UPDATE guild_member SET active_persona = NONE …` — no `DELETE FROM channel_active_persona`. The only two writers of `channel_active_persona` are `wear.rs:149` and `wear.rs:166`. `schema.surql:124-128` defines the table with NO cascade event. `messages/mod.rs:123-128` reads `active_persona` straight from `channel_active_persona` with no `can_edit_persona` check; `posting.rs:100` uses that `stored_persona` for the bare branch without re-deriving authz.
**Trigger.** (1) Owner shares persona P. (2) Editor wears P in channel C via `PUT /channels/{C}/active-persona`. (3) Owner revokes: `DELETE /personas/{P}/editors/{aid}`. (4) Ex-editor POSTs a bare message (no `persona_id`) to `/channels/{C}/messages` → message stamped `persona_id=P`, `persona_name=P.name`.
**Why it matters.** The per-channel wear path is the CURRENT path that supersedes per-guild wear. It lets a revoked editor continue authoring under a persona they no longer have any right to wear; authorization is NOT re-derived on the bare-message send for the stored-wear branch — it relies on revoke-time cleanup that misses `channel_active_persona`.
**Confidence:** 0.8. **Reachability:** confirmed.
**Anchor-test gap.** `redeem_grants_edit_and_wear_then_revoke` (personas.rs:413) wears only via the per-GUILD path and asserts only PATCH 404s after revoke; it never wears per-channel, revokes, then sends a bare message. No test cleans/asserts `channel_active_persona` on `remove_editor`.
**Suggested fix.** In `remove_editor`'s transaction add `DELETE FROM channel_active_persona WHERE persona = type::record('persona',$pid) AND account = type::record('account',$aid);`. Defense-in-depth: re-check `can_edit_persona` on the stored-wear branch in `posting.rs` before stamping, falling back to no-persona when the caller can no longer edit it.

#### F-D8-3 — `src/server/media.rs:101-110, 229-237` — inv10 — security
**Claim.** Upload performs NO MIME validation: `next_file_field` accepts any client-supplied `Content-Type` (only length-capped at 255 chars), and `serve_original` reflects that stored MIME verbatim on GET with no `X-Content-Type-Options: nosniff` and no `Content-Disposition`. An uploader can store `text/html` or `image/svg+xml` and have it served as active content on the app's own origin.
**Evidence.** `next_file_field` (lines 102-105) takes `field.content_type()` and stores it if `len<=255`, else octet-stream — no allowlist (no check that it is `image/*`). `serve_original` (lines 230-237) sets `CONTENT_TYPE` to the stored mime with no nosniff header and no `Content-Disposition: attachment`. No validation that the bytes match the declared mime.
**Trigger.** `POST /media` with multipart `Content-Type: text/html` and an HTML/JS payload → 201, row stored with `mime=text/html`. Any authed account then `GET /media/{id}` → response `Content-Type: text/html`, body executed in the browser at the app origin. Combined with the documented non-ACL (any authed account fetches any id, `tests/media.rs:125`) this is stored XSS reachable by any registered user against any other.
**Why it matters.** inv10 names "missing MIME validation". Serving attacker-controlled active content same-origin with no nosniff / no Content-Disposition is a stored-XSS / content-sniffing vector against a server-trusted multi-user app. The media store is meant for images but enforces nothing.
**Confidence:** 0.7. **Reachability:** confirmed.
**Anchor-test gap.** Tests assert MIME ROUND-TRIPS (`upload_then_download_round_trips_bytes_and_mime`, `thumbnail_query_on_non_image_falls_back_to_original`) — they LOCK the current behavior of reflecting arbitrary mime verbatim. No test asserts a nosniff header, a Content-Disposition, or rejection of non-image mimes; the safe behavior is NOT covered (the tests characterize the unsafe behavior as intended).
**Suggested fix.** Allowlist mimes to `image/{png,jpeg,webp,gif}` on upload (reject others, or sniff magic bytes); on GET always add `X-Content-Type-Options: nosniff` and, for non-image mimes, `Content-Disposition: attachment` so the browser never renders stored bytes as active content.

**Trigger.** Editor wears P via `PUT /channels/{C}/active-persona`, then `DELETE /personas/{P}/leave`, then POSTs a bare message to `/channels/{C}/messages` → still stamped as P though `can_edit_persona` would now return false.
**Why it matters.** Same inv3 violation as `remove_editor` but self-initiated; wear outlives editor status because the per-channel stored wear is neither gated at send time nor cleared on leave.
**Confidence:** 0.9. **Reachability:** confirmed.
**Anchor-test gap.** `owner_shares_with_friend_then_friend_leaves` (personas.rs:641) wears via the per-GUILD path before leaving and asserts only list membership; it never wears per-channel then leaves then sends a bare message.

#### F-D6-1 — `src/server/auth/registration.rs:190-213` — inv13 — concurrency
**Claim.** `create_account` issues a bare `CREATE account` against the `account_username_ci UNIQUE` index WITHOUT `with_write_conflict_retry`, yet the caller maps `is_unique_violation` → 409. Two concurrent `POST /auth/register` with the same username both pass validation and both CREATE; MVCC arbitrates. The loser can surface a *write-conflict* error (not the UNIQUE-violation text); with no retry wrapper that propagates as a raw `surrealdb::Error` → 500 instead of the intended idempotent 409.
**Evidence.** `registration.rs:196-209` calls `state.db.query("CREATE account SET …").await?.check()?` with no retry wrapper (the ONLY `is_unique_violation` consumer NOT in the retry caller set; cf. `membership.rs:136` which wraps the identical pattern and pre-checks). `schema.surql:25` `DEFINE INDEX account_username_ci ON account FIELDS username_ci UNIQUE`. The canary's own message (`retry_canary.rs:196-197`) lists "account" as a 409 path to audit, confirming design intent.
**Trigger.** Two simultaneous `POST /auth/register` with the same username (case-insensitive collapse). Under contention the loser can return "Write conflict, retry the transaction" rather than "already contains"; with no retry the handler returns 500.
**Why it matters.** Breaks inv13 for the account-creation path: a registration race intermittently 500s instead of returning the deterministic 409, and the transient write-conflict is never retried even though retrying against a fresh snapshot would surface the clean UNIQUE violation → 409.
**Confidence:** 0.6. **Reachability:** confirmed.
**Anchor-test gap.** `duplicate_username_is_409_case_insensitive` (tests/auth.rs:50) is SEQUENTIAL — it exercises steady-state UNIQUE 409 but never the concurrent race. No concurrent-register test exists.
**Suggested fix.** Wrap the CREATE in `with_write_conflict_retry(|| async { … })` exactly as `membership.rs:136` does, then let the residual `is_unique_violation` mapping at `registration.rs:52` produce the 409. Add a concurrent-register integration test (two `tokio::spawn`'d register calls, same username) asserting one 201 + one 409, never a 500.

#### F-D6-2 — `src/server/personas/wear.rs:144-174` — inv13 — concurrency
**Claim.** `set_channel_active_persona` runs a DELETE-then-CREATE on `channel_active_persona` inside one transaction, but the table has UNIQUE index `channel_active_persona_pair (account, channel)` and the CREATE is NOT wrapped in `with_write_conflict_retry`, NOR is any residual error mapped — every `Err` goes to 500. Two concurrent `PUT /channels/{cid}/active-persona` for the same `(account, channel)` race; the loser gets a write-conflict (or UNIQUE "already contains"), surfacing as 500.
**Evidence.** `wear.rs:148-162` `query("BEGIN; DELETE …; CREATE channel_active_persona SET …; COMMIT")` with no retry wrapper; the match at `175-181` sends every `Err` to `INTERNAL_SERVER_ERROR`. `schema.surql:128` defines the UNIQUE pair. Contrast `push.rs:180-182` which uses the same DELETE-then-CREATE idiom — both inherited it without the retry wrapper.
**Trigger.** Same authenticated user firing two `PUT /channels/{cid}/active-persona` simultaneously with `persona_id` set (realistic on double-tap / retry-on-slow-network). The loser conflicts on the UNIQUE pair → 500.
**Why it matters.** Breaks inv13 for the wear path: a benign concurrent re-wear that should be idempotent intermittently 500s. A retry would let the loser re-run against the winner's snapshot and converge.
**Confidence:** 0.7. **Reachability:** confirmed.
**Anchor-test gap.** No concurrent active-persona test exists; the write-conflict-on-`(account,channel)` branch is uncovered.
**Suggested fix.** Wrap the `Some(pid)` branch's transaction in `with_write_conflict_retry` (idempotent DELETE+CREATE, safe to retry). Add a concurrent-wear test asserting both requests resolve to 204 with exactly one surviving row.

#### F-D6-3 — `src/server/push.rs:183-207` — inv13 — concurrency
**Claim.** `store_subscription` does DELETE-then-CREATE on `push_subscription` keyed by the UNIQUE `endpoint` index, in one transaction, NOT wrapped in `with_write_conflict_retry`, and the caller routes every `Err` to 500. Two concurrent `POST /push/subscribe` with the same endpoint (browser re-subscribe firing twice) race; the loser hits write-conflict or "already contains" and returns 500 instead of the documented idempotent 204.
**Evidence.** `push.rs:191-198` BEGIN/DELETE/CREATE/COMMIT with no retry wrapper; `push.rs:172-176` maps any `Err` to `INTERNAL_SERVER_ERROR`. `schema.surql:226` `DEFINE INDEX push_subscription_endpoint … UNIQUE`. The module doc (`push.rs:180-182`) calls this an upsert that "mirrors the codebase's defensive-upsert idiom" — same un-retried idiom as `wear.rs`.
**Trigger.** A browser/service-worker re-subscribing twice in quick succession (same endpoint). Concurrent transactions conflict on the endpoint UNIQUE index; loser returns 500.
**Why it matters.** Breaks inv13 for push-subscribe; the suite asserts re-subscribe is idempotent 204 (push.rs:89-102) but only sequentially. A concurrent re-subscribe can 500 for a benign idempotent retry.
**Confidence:** 0.7. **Reachability:** confirmed.
**Anchor-test gap.** `tests/push.rs:89-102` asserts re-subscribe is 204 with one row but SEQUENTIALLY. No concurrent same-endpoint subscribe test; the write-conflict/UNIQUE-race branch on `endpoint` is uncovered.
**Suggested fix.** Wrap `store_subscription`'s transaction in `with_write_conflict_retry` (idempotent, safe). Add a concurrent same-endpoint subscribe test asserting both → 204 and exactly one row.

### LOW

#### F-D1a-1 — `src/server/permissions.rs:25-47` — inv2 / soft-delete read-write asymmetry — correctness
**Claim.** `caller_role` (and therefore `require_manager` / `require_owner` which build on it) reads `guild_member` with NO `guild.deleted_at = NONE` filter. Every manager/owner-gated mutation succeeds against a SOFT-DELETED guild: an owner/admin can `patch_guild`, `create_channel`, `patch_channel`, `delete_channel`, `restore_channel`, `invite_member`, `remove_member`, and `set_member_role` on a guild already trashed. Reads (`get_guild`, `list_members`, `load_guild_detail`) DO filter `deleted_at = NONE`, so the guild is invisible to reads but still fully mutable by its managers.
**Evidence.** `permissions.rs:34-44` query is `SELECT role FROM guild_member WHERE guild = … AND account = …` with no liveness clause. Contrast: `mod.rs:90-92` (load_my_guilds), `mod.rs:166-173` (my_guild_ids), `access.rs:62-67` (resolve_membership) all add the liveness filter. The channel/membership mutations gate ONLY on `require_manager`/`require_owner`/`caller_role` and none re-check liveness. `deletion.rs:62` even comments that `require_owner` "reads guild_member, which survives a soft-delete" — the author knows the membership row outlives the soft-delete, but the implication for management mutations is unguarded.
**Trigger.** Owner O creates guild G with channel C, calls `DELETE /guilds/{G}` (204), then `POST /guilds/{G}/channels` or `PATCH /guilds/{G}` or `POST /guilds/{G}/members` — each returns 2xx and writes to the trashed guild instead of 404/409.
**Why it matters.** Not a privilege-boundary crossing (the actor is the guild's own owner/admin) — hence low severity, no cross-tenant escalation. But it is a correctness defect against the soft-delete model (`schema.surql:97-99`, `deletion.rs:27-30` "It vanishes from every read"): writes land in an entity the rest of the system treats as gone. Side effects: orphaned channels/members created post-trash, confusing restore, invites that silently land users in a dead guild, and a wider window for the 30d purge to delete rows a manager just created.
**Confidence:** 0.6. **Reachability:** confirmed.
**Anchor-test gap.** `soft_delete.rs` covers soft-delete→read-hidden→restore but NO test attempts a manager mutation on an already-soft-deleted guild and asserts rejection. The "manager mutates a trashed guild" branch is untested.
**Suggested fix.** Option (b) preferred: add an explicit `guild_is_live(state, gid)` precheck after `require_manager` in `create_channel`/`patch_channel`/`delete_channel`/`restore_channel`/`patch_guild`/`invite_member`/`remove_member`/`set_member_role` (excluding `delete_guild`/`restore_guild`, which must legitimately operate on a trashed guild). Option (a) — adding the liveness filter to `caller_role` itself — breaks `restore_guild`.

#### F-D1b-3 — `src/server/personas/core.rs:383-391` — delete_persona cascade leaves no dangling worn-persona references — correctness
**Claim.** `delete_persona` cascades `persona_image`, `persona_editor`, and `guild_member.active_persona` but does NOT delete `channel_active_persona` rows pointing at the deleted persona. The dangling row means a bare message in that channel is still stamped with the now-deleted persona's id, producing a message attributed to a persona that no longer exists.
**Evidence.** `core.rs:383-391` deletes `persona_image`, `persona_editor`, updates `guild_member.active_persona=NONE`, deletes the persona — but no `DELETE FROM channel_active_persona`. `schema.surql` has no cascade event. `messages/mod.rs:123-128` reads `channel_active_persona` unconditionally; the name-join finds no persona, but `persona_id` is still emitted.
**Trigger.** Owner wears P via `PUT /channels/{C}/active-persona`, then `DELETE /personas/{P}`, then POSTs a bare message to `/channels/{C}/messages` → message carries `persona_id` of the deleted persona.
**Why it matters.** Not a cross-account authz break (only the owner reaches this), so data-integrity/orphan-row rather than escalation — but it leaves the per-channel wear table inconsistent with the deletion contract `delete_persona` otherwise upholds.
**Confidence:** 0.8. **Reachability:** confirmed.
**Anchor-test gap.** `deleting_persona_keeps_its_name_on_past_messages` (personas.rs:325) covers snapshot retention for an ALREADY-SENT message, not a NEW bare message sent after delete via a stale `channel_active_persona` row.
**Suggested fix.** Add `DELETE FROM channel_active_persona WHERE persona = type::record('persona',$pid);` to `delete_persona`'s transaction.

#### F-D4-3 — `src/server/messages/reading.rs:238-244` — inv7 (cursor input validation) — correctness
**Claim.** `is_rfc3339` is only a fixed-position separator probe; a cursor string with correct separators at positions 4/7/10/13/16 and length≥20 but a malformed sub-second/offset tail passes the 400 gate, then fails inside `type::datetime($cursor)` and bubbles to a 500 instead of a clean 400.
**Evidence.** `l.238-244` checks only `b[4]=='-'`, `b[7]=='-'`, `b[10]=='T'`, `b[13]==':'`, `b[16]==':'` and `len>=20`. A value like `2026-05-22T12:00:00Xbogusbogus` passes. `parse_cursor` gates on `is_rfc3339` only; the raw string is bound to `type::datetime($since/$before)` (reading.rs:340,352) which throws → `INTERNAL_SERVER_ERROR` (l.62-65) rather than the intended typed 400.
**Trigger.** `GET /channels/{cid}/messages?before=2026-05-22T12:00:00Xbogus&before_id=abc`. The doc comment at l.236-237 states the probe's purpose is exactly to avoid a 500 — this input defeats it.
**Why it matters.** Self-described contract of the probe is not fully met. Minor robustness/observability issue (500 vs 400, plus an error log per malformed poll).
**Confidence:** 0.78. **Reachability:** confirmed.
**Anchor-test gap.** No test sends a malformed-tail cursor and asserts a 400 (it currently 500s).
**Suggested fix.** Replace the positional probe with `chrono::DateTime::parse_from_rfc3339(s).is_ok()`, or parse to a typed `Datetime` in `parse_cursor` and bind that, so any malformed cursor yields a deterministic 400.

#### F-D7-1 — `src/server/mod.rs:248-274` — inv14 — correctness
**Claim.** The guild purge cascade orphans child rows in tables it does not delete: a guild purged at 30d removes guild/channel/guild_member/message but leaves `custom_emoji` (guild FK), `user_guild_order` (guild FK), `lorebook_entry` (channel FK) and `channel_active_persona` (channel FK) pointing at now-nonexistent records. inv14 says the purge "cascades the channels/members/messages of a purged guild" — these four are real per-guild/per-channel children with required `record<>` FKs and are never swept.
**Evidence.** `purge_soft_deleted` only issues DELETE against message, channel, guild_member, guild (`mod.rs:258-268`). `schema.surql` shows `custom_emoji.guild` (235) and `user_guild_order.guild` (250) are `TYPE record<guild>`; `lorebook_entry.channel` (178) and `channel_active_persona.channel` (126) are `TYPE record<channel>` — all required FKs. No `DEFINE EVENT`/`ON DELETE` cascade exists (grep clean).
**Trigger.** Create a guild + custom emoji and/or lorebook entry and/or channel active persona and/or rail-order row; `DELETE /guilds/{id}`; backdate `deleted_at` past 30d; run purge. The guild and channels are hard-deleted but the four child rows survive as dangling-FK orphans.
**Why it matters.** Orphan rows with required `record<>` FKs accumulate forever (purge is the only GC) and point at deleted parents — storage leaks, and stale data surfaces via any query reading these tables by account rather than by live guild/channel. Whether this strictly violates inv14 depends on reading its child-list as exhaustive vs minimal — flagged for adjudication.
**Confidence:** 0.6. **Reachability:** confirmed.
**Anchor-test gap.** `soft_delete.rs` asserts cascade only for channel, message, guild_member. No test seeds or asserts cleanup of `custom_emoji`, `user_guild_order`, `lorebook_entry` or `channel_active_persona` across a purge.
**Suggested fix.** Extend `purge_soft_deleted` to also `DELETE custom_emoji WHERE guild IN $g`; `DELETE user_guild_order WHERE guild IN $g`; `DELETE lorebook_entry WHERE channel IN (SELECT VALUE id FROM channel WHERE guild IN $g)`; `DELETE channel_active_persona WHERE channel IN (…)`. Mirror the `guild_member` inline-subquery workaround where composite indexes are involved. Add a `soft_delete.rs` test seeding each child table.

#### F-D7-2 — `src/server/mod.rs:259-261` — inv14 — correctness
**Claim.** The channel 1d purge path cascades only the channel's messages; it does not delete `lorebook_entry` or `channel_active_persona` rows of a channel purged at 1d (without any guild involvement). A standalone soft-deleted channel that ages past 1d leaves its lorebook entries / active-persona rows orphaned.
**Evidence.** `mod.rs:259-261` deletes message rows of soft-deleted channels then the channels themselves, but nothing deletes `lorebook_entry` (`schema.surql:178`, channel FK) or `channel_active_persona` (`schema.surql:126`, channel FK) for those channels. A lorebook channel stores its entries here. The channel-purge test seeds only a message.
**Trigger.** Create a lorebook channel, add a lorebook entry, `DELETE /guilds/{id}/channels/{cid}`, backdate `deleted_at` past 1d, run purge. Channel row is gone; the `lorebook_entry` row survives with a dangling channel FK.
**Why it matters.** Same orphan class as the guild finding but reachable via the independent channel-only purge path, so it needs its own fix even if the guild cascade is fixed.
**Confidence:** 0.7. **Reachability:** confirmed.
**Anchor-test gap.** `purge_cascades_channel_to_its_messages` asserts only that the channel's message is hard-deleted; no assertion on `lorebook_entry` or `channel_active_persona` for a purged channel.
**Suggested fix.** In the 1d channel-purge block, before deleting channels, add `DELETE lorebook_entry WHERE channel IN (SELECT VALUE id FROM channel WHERE deleted_at != NONE AND deleted_at < time::now() - 1d);` and the same for `channel_active_persona`.

#### F-D12-4 — `src/server/messages/posting.rs:36-53` — attachment cap bounds dedup work — quality
**Claim.** Attachment dedup runs an O(n²) linear-scan (`attachments.contains(&id)`) over the fully-untrusted `attachment_ids` vector BEFORE the `MAX_ATTACHMENTS` cap is checked, so the cap does not bound the dedup work — only the 512 KiB JSON body limit does.
**Evidence.** `posting.rs:37-41` loops over `req.attachment_ids` calling `attachments.contains(&id)` (linear in current len); the `attachments.len() > MAX_ATTACHMENTS` cap check is at line 51, AFTER the loop. `tests/messages.rs:521` comment confirms "The handler dedupes attachments before counting". With short ids (~10-20 bytes each), ~10k-20k distinct ids fit in 512 KiB → ~10⁸ `contains()` comparisons before the cap rejects.
**Trigger.** `POST /channels/{cid}/messages` with `attachment_ids` containing ~15,000 distinct short strings (under the 512 KiB body limit). The dedup loop performs O(n²) comparisons before line 51 rejects.
**Why it matters.** A single authenticated member can spend server CPU on quadratic dedup per request before the cap fires; repeated, a low-grade CPU-amplification DoS. Bounded by the 512 KiB body limit so not catastrophic, but the cap meant to bound this work is checked too late.
**Confidence:** 0.6. **Reachability:** confirmed.
**Anchor-test gap.** `attachment_cap_boundary_is_100` sends exactly 101 ids; nothing exercises a large pre-cap array, and there is no early `len()` guard to test.
**Suggested fix.** Check `req.attachment_ids.len() > MAX_ATTACHMENTS` (or a hard upper bound) before the dedup loop, and/or use a `HashSet` for O(1) membership so dedup is O(n).

#### F-D13-1 — `src/ui/shell/channel/mod.rs:868-919` — autocomplete-owns-Enter contract (and the IME/send-path invariant) — docs-mismatch
**Claim.** The composer keydown comment (line 873) states the open autocomplete popover "owns the arrow/Enter/Tab/Escape keys", but the match arms (876-903) handle only ArrowDown/ArrowUp/Escape/Tab — there is NO Enter arm. With the popover open and a non-empty suggestion list, pressing Enter falls through to line 909 and SENDS the message with the literal `:query` text instead of accepting the highlighted suggestion.

Enter is not intercepted while `ac_token` is `Some`, so the guard at 909 fires, calling `act::send_message(s)` then clearing `ac_token` (918). #### F-D6-5 — `src/server/retry.rs:10-19, 36-47` — n/a (doc) — docs-mismatch

**F-D8-2** — `media.rs:247-260` (`make_thumb` calls `image::load_from_memory` with no decode limits → decompression-bomb OOM). *Refute:* The finder missed the guard built into the `image` crate's default decode path. `load_from_memory` → `ImageReader` sets `limits: Limits::default()` which carries `max_alloc: Some(512 MiB)`; `.decode()` calls `limits.reserve(decoder.total_bytes())?` BEFORE allocating the pixel buffer, where `total_bytes` is computed from HEADER-declared dimensions. A 60000×60000 RGBA bomb (~14.4 GB) fails at `reserve()` with `ImageError::Limits` BEFORE any multi-GB allocation; `make_thumb` propagates the Err and the handler gracefully serves the original (media.rs:207-208). Worst-case allocation that passes the 512 MiB cap is ~134 MP, bounded and per-decode. The "no Limits configured" evidence is the missed guard — the default IS applied. Hardening, not a defect.

**F-D8-4** — `media.rs:42-78` (no per-blob content-size cap beyond 64 MiB transport limit; full in-memory buffering). *Refute:* Code claims accurate, but the load-bearing premise is false: the finding claims "inv10 lists 'missing size validation'." The actual inv10 (ARCHITECTURE.md:382-384) is titled "Media path-traversal" and concerns ONLY random names + canonicalize + starts_with — nothing about size. No documented invariant requires a per-blob size cap distinct from the transport limit; the 64 MiB cap is a deliberate documented choice (raised to support multi-MB phone photos). Finder conceded "not strictly absent", confidence 0.3. Missing-hardening on an auth-gated server-trusted endpoint; the genuine inv10 (traversal) IS anchor-tested.

**F-D8-5** — `media.rs:177-222` (canonicalize→read TOCTOU window). *Refute:* The stat-then-open gap exists mechanically, but the trigger is not route-reachable: `storage_path` is server-minted (`random_media_id` = 16 random bytes hex, written via bound params), no user input ever touches the path, and `media_dir` is server-owned with 128-bit-unguessable filenames. The only write API writes a fresh random name and offers no way to place a symlink or write an attacker-chosen name. Swapping a guarded file for a symlink in the sub-millisecond window requires an attacker who can already write into `media_dir` — a precondition that does not exist for any remote actor. The literal pattern inv10 names, but no reachable trigger.
