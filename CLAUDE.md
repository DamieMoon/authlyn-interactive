## Identity
A Discord-style roleplay chat app: accounts/sessions, guilds, channels, messages, personas (editors + per-channel wear), media uploads, and Web Push.

## Stack
- Single Rust crate, **Leptos 0.8 full-stack**: axum 0.8 SSR server + WASM (`hydrate()`) browser bundle from one `src/` tree. `crate-type = ["cdylib","rlib"]`.
- **SurrealDB 3.x** over WebSocket (Rust SDK pinned `=3.1.0-beta.3`, Cargo.toml); 18 SCHEMAFULL tables in `src/storage/schema.surql`, embedded via `include_str!`.
- Built with **cargo-leptos** (config in `[package.metadata.leptos]`). Key crates: leptos_axum, tokio, argon2 (argon2id), image (thumbnails), web-push (VAPID), gloo-net/web-sys (client), rmcp/reqwest (nova), freya/rfd (native `authlyn-native` client).
- Toolchain pinned in `rust-toolchain.toml`: stable + rustfmt + clippy + `wasm32-unknown-unknown`.

## Architecture
- Entry points: server `src/main.rs` (cfg `ssr`, `#[tokio::main]`); WASM `hydrate()` in `src/lib.rs` (cfg `hydrate`). Routing shell `src/app.rs`.
- Four disjoint feature graphs: `ssr` (server: axum/surrealdb/tokio/argon2/image/web-push), `hydrate` (browser: web-sys/gloo-net), `nova` (standalone MCP bridge `src/bin/nova-mcp.rs`, `required-features=["nova"]`, never in the Leptos build), `freya` (standalone native desktop/Android client `src/bin/authlyn-native.rs`, `required-features=["freya"]`, Skia/winit GUI + reqwest HTTP client, never in the Leptos build).
- Layout: shared always-on `protocol.rs` (serde wire DTOs), `markup/` (roleplay rich-text parser), `app.rs`, `ui/`; ssr-only `server/<domain>/`, `db.rs`, `storage/`; hydrate-only `client/api.rs`; freya-only `native/` (own reqwest `api.rs` + Freya `ui.rs`/`act.rs`/`state.rs`, reusing `protocol`/`markup`). Newcomer path: `app.rs` → `ui/shell/` → `client/api.rs` or `ui/shell/act/` → `protocol.rs` → `server/<domain>/` → `storage/schema.surql`.
- **Disjointness rule:** shared/WASM-bound modules import ZERO ssr crates (surrealdb/axum/tower); the hydrate bundle must stay free of ssr-only crates. All ssr/hydrate deps are `optional = true`, pulled only by their feature. `act/` submodules pair a hydrate-real impl with an ssr no-op stub so views call `act::xxx` ungated. The `freya` native client (`src/native/`) likewise imports ZERO ssr/hydrate crates; it's a true HTTP client reusing only the always-on `protocol`/`markup` modules (like `nova`).
- Real-time is **SSE (`GET /events`, tokio broadcast hub in `AppState.events`, id-only notify-and-fetch events filtered per-connection by `access::visible_channels`) with automatic client fallback to the legacy poll loop** (`act/sync.rs` drives; `act/message.rs::start_poll` is the fallback after 5 failed connects). Typing remains in-memory (`AppState.typing`), broadcast on POST. Batched `GET /unread` replaces per-channel unread probes. NOT LIVE SELECT (which exists only in tests).

## Invariants you must not break (cited to current code)
- Server identity comes ONLY from the `authlyn_session` cookie, never the request body (`server/auth/session.rs` AuthAccount extractor); DB stores only the SHA-256 of the token.
- Authorization is re-derived on every mutate: guild role from `guild_member` (`permissions.rs` caller_role) + channel membership + persona ownership; never trust stored state.
- Unauthorized collapses to a privacy-404 with an identical body — never reveal existence (`server/access.rs` resolve_membership).
- SQL is parameterized only: user VALUES via `.bind()` / `type::record(...)`; only compile-time consts (`MSG_PROJECTION`), loop indices, or static column fragments are ever spliced into query text (`messages/reading.rs`).
- Media: server-minted random 16-byte id for the on-disk path (`media.rs` random_media_id); GET canonicalizes + `starts_with(media_dir)` (`media.rs` serve_original); image-only MIME allowlist + `X-Content-Type-Options: nosniff`.
- The markup parser must stay panic-free on arbitrary input (`src/markup/tree.rs` — Root frame is never popped/closed).
- Message cursor is composite `(sent_at, id_key)` with strict tie-break; bind datetimes via `type::datetime($p)` — never `<string>`-cast a datetime feeding ORDER BY (`reading.rs` cursor branch; schema header warns of lex-misorder).
- Persona send-path: re-check `can_edit_persona` for BOTH suggested and stored persona on every send (`messages/posting.rs` `resolve_send_persona` — shared by the roll endpoint).
- Message effects (W4): `message.effect` is `option<string>`, server-validated against `MESSAGE_EFFECTS` (`whisper`/`shout`/`spell`; unknown → 400, `posting.rs`). Whisper bodies are masked to the fixed `(whisper)` placeholder in BOTH reply-quote snippets (`reading.rs` MSG_PROJECTION) and Web Push payloads (`push.rs` notification_body) — extend the mask to any NEW body-preview surface.
- Fate Engine (W4): `kind='roll'` messages are server-authoritative (RNG in `messages/rolling.rs`, grammar-bounded, persona double-checked via posting's path) and IMMUTABLE — author edit AND delete → 403 (`editing.rs`). System-message immutability is an authorship side-effect that does NOT transfer to roller-authored rows; the roll guard is explicit. Client affordances flow from the shared `message_actions(kind, mine)` predicate (`ui/shell/channel/mod.rs`) — never re-branch kind checks per surface.
- Ghost Quill (W4): typing-draft TEXT never rides the SSE bus (id-only invariant) — the `Typing` event only nudges a permission-checked `GET /channels/{cid}/typing-drafts` over an ephemeral in-memory map (`AppState.typing_drafts`, 8s TTL injectable via `with_draft_ttl`, mutex never held across await, cleared on send/roll). Opt-in BOTH ways: the sender's ping carries the draft only when their pref is on; the receiver fetches/renders only when theirs is.
- Soft-delete hides on read AND is immutable to management mutations (`permissions.rs` ensure_guild_live, called by require_manager).
- Persona identity (name/description/color/avatar) is SNAPSHOTTED onto the message row at send; never resolve it live (`posting.rs` persist_message).
- Racy CREATE on a UNIQUE index → `with_write_conflict_retry`, then 409/idempotent, never 500 (`src/server/retry.rs`).
- `is_admin` is fail-closed: empty admin set authorizes no one (`permissions.rs`). Purge cascades to children (`server/mod.rs` purge_soft_deleted).
- SCHEMAFULL NONE-coercion: a field added to a populated table must be `option<>` or get an idempotent UPDATE backfill BEFORE any other UPDATE, or schema apply crash-loops boot. `message.kind` (the system-vs-user discriminator) is the worked example: it is materialised INSIDE the existing first `message` backfill (`schema.surql`, alongside `attachments`/`pinged_users`, coalesced `?? 'user'`), NOT a separate statement — a separate one would re-validate the row while the arrays are still NONE. Re-splitting it crash-loops apply; `tests/schema_apply.rs::applying_kind_over_populated_messages…` guards it. (Data-safety lesson: ctx · learnings · I15.)
- App-admin system broadcast (Nova DOT): `POST /admin/system-message` (admin-gated, `server/system_messages.rs`) posts a `kind='system'` message into every live guild's first text channel, authored by the reserved seeded bot account `account:nova_dot` (login disabled via a non-PHC `password_hash` sentinel; reserves the `nova-dot` handle). The fan-out core is tested directly (the admin-allowed HTTP path races the `is_admin` env, per `tests/feedback.rs`'s convention). `MeResponse.is_admin` (on `/auth/me`) gates the composer UI in the account modal.
- `record<...>` links are type annotations only — SurrealDB does NOT enforce referential existence; cleanup/cascade is the app's job.
- Widening an enum ASSERT on an EXISTING field needs `DEFINE FIELD OVERWRITE` — `IF NOT EXISTS` never re-applies over a deployed field, so the old ASSERT silently survives and rejects the new value (found live when `message.kind` gained `'roll'`; prod-shaped guard `tests/schema_apply.rs::widened_kind_assert_reaches_a_db_where_kind_already_exists`). `message.kind` is the documented OVERWRITE exception in `schema.surql`'s header; `channel.kind`, `guild_member.role`, `message.effect`, `friendship.state` still use `IF NOT EXISTS` and need the same treatment if ever widened. OVERWRITE does NOT re-validate existing rows (empirically pinned by that test).

- Safari/WebKit drops the `Secure` session cookie over `http://localhost` (Chromium accepts it as a localhost exception). So local UI testing on WebKit/iOS over http "logs in" (`POST /auth/login` → 200) but never holds a session (`/auth/me` stays 401, app sticks on `/login`). This is correct prod behavior (prod is https) — `.secure(true)` in `session.rs` `session_cookie` stays. For local Safari/iOS testing, inject the `authlyn_session` cookie with `secure:false` (e.g. Playwright `context.addCookies`).
- The native client mirrors that same Secure-cookie trap: reqwest's cookie jar will NOT replay the `Secure` `authlyn_session` cookie over dev `http://`, so `src/native/api.rs` captures the token from login's `Set-Cookie` and attaches `Cookie: authlyn_session=<token>` manually on every request (`SESSION_COOKIE` const + `cookie_header`). The server `.secure(true)` stays; this is a client-side workaround, no backend change.
- The native (`freya`) graph has NO tests and NO CI coverage — `cargo test --features ssr` doesn't touch it, and no `.github/workflows/` job builds it. The only checks are a manual `cargo build --bin authlyn-native --features freya` + `cargo clippy --features freya`. `freya` is pinned exact (`=0.4.0-rc.21`) because rc churn is real (it dropped Dioxus at 0.4) — see the Cargo.toml comment before bumping.
- Mobile PWA safe-area: the installed app reserves notch / Dynamic Island / home-indicator space via `viewport-fit=cover` (`src/app.rs` viewport meta) + `env(safe-area-inset-*, 0px)` paddings on `.topbar` (`_content.scss`), `.app` left/right (`_layout.scss`), and — since the W3 hybrid-nav redesign — the fixed `.bottom-tabs` bar (`_nav.scss`), which OWNS the bottom inset (`--tabbar-h` knob + `env(safe-area-inset-bottom)`); the mobile composer drops its own bottom inset and sits on the tab bar's top edge, so the inset is counted exactly once. (The old edge-swipe drawer with its fixed `.rail`/`.sidebar` insets in `_mobile.scss` was deleted in W3/T4.) `env()` is 0 on non-notched devices, so it's a no-op there — but DON'T drop `viewport-fit=cover` or the insets all collapse to 0.
